Neovim Configuration

Tutorial

A comprehensive guide to your development environment

Welcome slide. This tutorial covers all the installed plugins and their use cases. Audience should already be familiar with basic vim concepts (hjkl, modes, basic editing).

This config is designed for multi-language development. Each quadrant represents a major feature category we'll cover.

Quick Start

git clone [repo] ~/.config/dev-env
cd ~/.config/dev-env/vim
make install

Prerequisites

  • neovim (0.9+)
  • node.js (for coc.nvim extensions)
  • ripgrep (for fast grep search)
  • Nerd Font (optional, for icons)
The Makefile handles symlinking configs and installing plugins. Node.js is required because coc.nvim extensions are npm packages. Ripgrep is used by fzf for blazing fast project searches.

Architecture

vim-plug Plugin manager
coc.nvim LSP client (completion, navigation, refactoring)
ALE Linting & formatting (eslint, prettier, gofmt)
Lua plugins Modern UI (lualine, nvim-tree, gitsigns)
This config uses a mix of VimScript and Lua plugins. coc.nvim handles LSP - this is where IDE features come from. ALE handles linting separately to avoid conflicts with coc. Lua plugins provide modern UI elements with better performance.

The leader key is the starting point for all custom mappings. We use comma because it's easy to reach and rarely used for editing. LocalLeader (double comma) is for language-specific commands like running tests.

which-key is your built-in cheat sheet. When you press leader and pause, it shows all available keybindings with descriptions. This is extremely helpful when learning the config.

coc.nvim

Conquer of Completion

Full LSP client providing IDE-like features

Installed Extensions

coc-tsserver - TypeScript/JS
coc-go - Go
coc-rust-analyzer - Rust
coc-solargraph - Ruby
coc-json - JSON
coc-eslint - ESLint
coc.nvim is our main LSP client. It handles all smart code intelligence. Each language has a specific extension that talks to its language server. ALE is disabled for LSP features to avoid conflicts.

Primary Features

KeyAction
Tab / Shift-Tab Navigate completion menu
Enter Confirm completion
Ctrl-Space Trigger completion manually
K Hover documentation
,gd Go to definition
,gr Find references

These are the core keybindings you'll use every day. K shows documentation in a floating window - super useful for API exploration. ,gd is your main navigation tool. Use Ctrl-o to jump back.

Code Actions & Diagnostics

KeyAction
,ca Code actions (auto-import, quick fixes)
,qf Quick fix current issue
[g / ]g Jump to prev/next diagnostic
,cf Format code
Diagnostics show inline with virtual text
Code actions are context-aware suggestions from the language server. Common uses: adding missing imports, implementing interfaces, extracting functions. [g and ]g navigate through errors from LSP (not ALE linter).

Exercise 2: LSP Navigation

  1. Open a code file (Go, TypeScript, Rust, or Ruby)

  4. Press Ctrl-o to jump back
  5. Press K to see documentation
  6. Press ,gr to find all references
  7. Try ,rn to rename a variable
This exercise gets students comfortable with core LSP features. The rename feature is particularly impressive to demonstrate LSP power.

ALE

Asynchronous Lint Engine

  • Runs linters in background without blocking
  • Auto-fixes code on save
  • Separate from coc.nvim (LSP disabled)
Division of Labor:
coc.nvim → LSP features
ALE → Linting & formatting
ALE complements coc.nvim rather than competing with it. We've explicitly disabled ALE's LSP features to avoid conflicts. Fix-on-save is enabled globally.

Primary Features

KeyAction
[e / ]e Jump to prev/next lint error
,d Show error detail
(automatic) Fix on save
Key Difference:
[e/]e - ALE linting errors (eslint, rubocop)
[g/]g - coc.nvim LSP diagnostics
The [e/]e vs [g/]g distinction is important but subtle. ALE errors come from dedicated linters. coc diagnostics come from language servers. On save, ALE runs all fixers automatically.

fzf.vim

Fuzzy Finder

  • Lightning-fast fuzzy file finding
  • Uses ripgrep for content search
  • Preview window on the right
  • Works with git to respect .gitignore
fzf is probably the tool you'll use most often. It's incredibly fast even on large codebases. The preview window lets you see file contents before opening.

Primary Features

KeyActionWhen to Use
,f Find files Open any file by name
,g Grep (ripgrep) Search text in codebase
,b List buffers Switch open files
,* Grep word under cursor Find usages
These four commands cover 90% of navigation needs. ,f is for when you know the filename. ,g is for when you know the content. ,* is a quick way to find all usages of something.

Git-Aware Search

KeyAction
,gf Git files (respects .gitignore)
,gs Git status files (modified only)
Tip: Use ,gs to quickly jump between files you're currently editing
Git-aware search is great for focusing on relevant files. ,gf excludes node_modules, build folders, etc. ,gs is perfect when working on a feature across multiple files.

Git Integration

gitsigns.nvim

KeyAction
]c / [c Jump to next/prev hunk
,hp Preview hunk (see diff)
,hb Blame current line
Sign Column: │ add/change   _ delete
gitsigns shows git status in the sign column. Hunks are chunks of changed code. ,hp shows a floating window with the diff for that hunk. ,hb shows who last modified the current line.

vim-fugitive

Git Commands

:Git / :G Open Git status
:Git blame Full file blame
:Git diff Diff view
:Gwrite Stage current file
:Gread Checkout current file
Fugitive is the definitive Git plugin for Vim. :G opens an interactive status view where you can stage with s. :Git blame opens a side-by-side blame view.

vim-rhubarb

GitHub Integration

:GBrowse

Opens current file on GitHub

Works with visual selection for line ranges!
Select lines, then :GBrowse to share a permalink.
:GBrowse is incredibly useful for sharing code with teammates. Select lines in visual mode and :GBrowse creates a permalink. Also works for PRs, issues, and commits.

Exercise 6: Git Workflow

  1. Make a small change to a file
  2. Use ]c to jump to the change
  3. Press ,hp to preview the hunk
  4. Press ,hb to see line blame
  5. Run :Git blame for full file blame
  6. Run :G to see git status
This exercise covers the most common git operations. Students should feel comfortable navigating changes and viewing history.

lualine provides at-a-glance status information. The mode indicator changes color based on current mode. Git diff stats show how many lines are added/changed/removed. Diagnostics show error and warning counts from LSP.

vim-commentary is incredibly useful for quick commenting. gc works like any vim operator - combine with motions. gcap comments the current paragraph (blank-line delimited). Works in visual mode for arbitrary selections.

vim-surround

Surround Text

KeyActionExample
cs"' Change " to ' "hello" → 'hello'
ds" Delete " "hello" → hello
ysiw" Surround word hello → "hello"
S" Surround selection Visual + S"
Works with: " ' ` ( ) [ ] { } < > t (tags)
vim-surround is essential for editing quotes, brackets, tags. cs = change surrounding, ds = delete surrounding, ys = you surround. The 't' target works with HTML/XML tags. S in visual mode wraps the selection.

indent-blankline.nvim

Indent Guides

Without

function example() {
  if (condition) {
    for (i = 0; i < n; i++) {
      doSomething();
    }
  }
}

With Guides

function example() {
│ if (condition) {
│ │ for (i = 0; i < n; i++) {
│ │ │ doSomething();
│ │ }
│ }
}
  • Shows vertical indent guides
  • Highlights current scope
  • Helps track nesting level
indent-blankline makes code structure visible. The current scope is highlighted more prominently. Essential for deeply nested code in Python, YAML, etc.

Window Navigation

KeyAction
Ctrl-h / j / k / l Move between windows
Ctrl-d / Ctrl-u Half-page scroll (centered)
n / N Search results (centered)
Centered scrolling: The cursor stays in the middle of the screen when jumping
Ctrl-hjkl replaces the longer Ctrl-w hjkl for window navigation. Centering after scrolls and searches reduces disorientation. These small improvements add up to a smoother experience.

Buffer Management

KeyAction
,n / ,p Next/previous buffer
,bd Delete buffer
,b List buffers (fzf)
Quick shortcuts:
,w write    ,q quit
Buffers are like tabs - they hold open files. ,n and ,p cycle through open buffers quickly. ,b opens fzf to fuzzy-search through buffers. ,bd closes the current buffer without closing the window.

Standard vim split commands still work. Ctrl-w prefix is for window operations. Use splits when comparing files or working on related code.

Troubleshooting

Common Commands

CommandPurpose
:CocInfo Check LSP status
:CocRestart Restart language server
:ALEInfo Check linter status
:PlugStatus Check plugin status
:checkhealth Neovim health check
These are your first stop when something isn't working. CocInfo shows which language servers are running. CocRestart often fixes completion issues. checkhealth reveals missing system dependencies.

Common Issues

Completions not working? 
:CocRestart
Not auto-formatting on save? 
:ALEInfo

Check if fixers are configured for this filetype

Plugin not loading? 
:PlugStatus
:PlugInstall
Most completion issues are resolved by restarting coc. ALE handles formatting - check ALEInfo if files aren't being formatted. PlugStatus shows red for failed plugins.
